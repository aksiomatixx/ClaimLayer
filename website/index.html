What this artifact demonstrates

ClaimLayer is my application for AI deployment, forward-deployed, and AI product roles — in the form of running software instead of bullet points. Each claim below is backed by code you can inspect, tests you can run, and a demo you can click through.

🎯

Problem selection

Every AI vendor in claims sells prediction — scoring which claims will go sideways. The pain I lived for ten years was carrying out the workflow. I built for the pain: agents that execute the regulatory process itself, not a dashboard that scores it.

🧭

Scoping judgment

One jurisdiction (California), one wedge (the adjuster's decision surface), worker and employer portals deliberately demoted to supporting cast. Depth over breadth is a product decision, and this site documents why I made it.

🤝

Trust-boundary design

The core craft of AI deployment: deciding, action by action, what the model may do alone, what it may only suggest, and what it must never touch. ClaimLayer's decision register below is that map, with the trade-offs stated.

🛡️

Guardrails as engineering

Hard limits live in the service layer, not the prompt. The model cannot deny treatment, move a statutory deadline, write a reserve, or invent a statutory value — regardless of what it returns, and regardless of any prompt injection.

📊

Observability & evals

Every model call is logged with its input snapshot, raw output, tokens, latency, guardrails triggered, and human-override status — queryable in an in-app console. The test suite adversarially tries to push agents past their bounds and asserts the guardrails hold.

🚢

Shipping, end to end

1,312 automated tests across 91 suites, CI that proves every migration applies to a clean database, 75+ reviewed pull requests, server-side PDF generation, webhook integrations, and a deployed public demo — designed and delivered by one person directing AI.

Case study · 01 — the problem

The problem, as lived — not as researched

Traditional adjusting is reactive. Medical reports, work status reports, and legal documents arrive all day; the job is to notice them, read them, file them, and work out what each one requires — while statutory clocks run and penalties accrue for missed deadlines. I didn't interview users to learn this. I was the user, for a decade.

The day I lived — react to paperwork

Medical reports pile into an inbox · work status reports wait to be read · legal documents get filed by hand · the adjuster works out what each one requires · notices and diaries typed one at a time · deadlines tracked in heads and spreadsheets.

The day I built — decide, then it executes

Every document ingested, classified, summarized, and reviewed in the context of its claim · required actions surfaced on a ranked queue with deadlines computed · the adjuster makes the call · the AI handles everything after: decision documented, notices sent, diaries completed and reset, state reports filed — all auditable.

The product insight: don't automate the decision — automate everything around it. Inbound is automated, outbound is automated, and the licensed human does exactly what the license is for: deciding. As long as the adjuster works the queue, statutory timelines are met, penalties are avoided, and the state reporting files itself.
Case study · 02 — the system

What the agents actually do

Six specialized Claude agents, each with authored prompts, explicit invocation triggers, and bounded output schemas — running on top of a customer's retained system of record through a pluggable integration layer. A layer, not a replacement: adoption was a design constraint, not an afterthought.

AI Agent

Compensability analysis

Score, priority, suggested reserves, red flags, and written rationale on every new claim — presented as a plain-language decision brief with source documents one click away. Accept or deny stays a licensed-human action.

AI Agent

MTUS treatment authorization

Every RFA reviewed against California treatment guidelines. The agent can only return auto_approve or physician_review — denial doesn't exist in its output schema.

AI Agent

Settlement pricing & MSA screening

C&R offers priced with written rationale, gated by a deterministic Medicare-interest screen that runs before the model ever sees the file.

AI Agent

Document classification & intake

Real PDFs — uploads, scanned faxes, email attachments — classified into a controlled category list, matched to their claim, summarized, and translated by a deterministic rules table into the action they require. Low confidence routes to a human triage queue, never a silent filing.

Engine

Statutory benefits & deadlines

TD at two-thirds AWW with the 104-week cap, PD rating and advances, calendar- vs business-day deadlines per code section, automated FROI/SROI state reporting — all deterministic, version-pinned to DWC sources, never model-generated.

Trust

AI decision audit trail

Every model call logged with its input snapshot, raw output, tokens, latency, guardrails triggered, and human-override status — queryable in the in-app Agents console. The audit trail is a product surface, not a log file.

Case study · 03 — the decision register

The calls I made, and what each one cost

This is the part of AI product management that doesn't show up in a feature list: deciding where the model is trusted, where it isn't, and what you give up by drawing the line where you drew it. Every decision below is enforced in code and covered by tests.

Product strategy
01
Execution over predictionThe market scores claims; ClaimLayer carries out the regulatory workflow itself — drafting analyses, applying statutory math, generating notices, tracking deadlines.Trade-offExecution demands regulatory correctness, which is far harder to build and test than a risk score — that difficulty is the moat.
02
A layer, not a replacementPluggable adapters sit on top of the retained system of record — ingesting claims, pushing diaries, documents, and notices back. No migration project required.Trade-offIntegration surface area and idempotency complexity, accepted because rip-and-replace is why claims systems never get modernized.
03
One jurisdiction, one decision surfaceCalifornia only; the adjuster's queue and drawer as the product wedge. Worker and employer self-service exist in the app but are deliberately not the pitch.Trade-offA smaller story to tell, in exchange for statutory depth no horizontal product can match.
04
Automate around the decision, not the decisionInbound documents and outbound aftermath are fully automated; every consequential call — compensability, authorization, settlement — is a licensed human's, made from a prepared brief.Trade-offA lower headline automation rate than "AI decides your claim," in exchange for a system a regulated carrier could actually adopt.
Trust boundaries — enforced in code, not prompts
05
No auto-deny pathway exists anywhere in the systemOn treatment authorizations the agent's output schema contains only auto_approve and physician_review. Denial is a licensed-human-only action by construction.Trade-offSome automatable denials wait for a human — the cost of making the system safe to point at people's medical care.
06
Statutory values are never model-generatedRating schedules, fee schedules, and caps come from version-controlled DWC publications. The model reasons over them; it never invents them.Trade-offA data-maintenance obligation when the state publishes updates, instead of a hallucination risk on benefit math.
07
Reserve changes require a licensed adjuster's approvalThe AI may suggest reserves; nothing writes to the financial system of record until a human approves it.Trade-offAn approval step on every financial write, because a wrong automated reserve is a real-money error with audit consequences.
08
Deterministic gates in front of the modelA rules-based MSA screen runs before settlement pricing; a deterministic rules table — not the model — translates classified documents into required actions; penalty diaries cannot be snoozed and refuse audited edits.Trade-offLess model autonomy where it would be most impressive, and most dangerous.
09
Low confidence routes to humans, never to silenceA document the classifier can't confidently match goes to a human triage queue. Nothing is ever silently filed.Trade-offA triage queue someone must work, instead of misfiled documents no one knows about.
10
Every model call is audited, and the audit is adversarially testedInput snapshot, output, guardrails triggered, human overrides — all queryable in-app. The test suite actively attempts to push agents past their bounds and asserts the guardrails hold.Trade-offLogging and storage overhead on every call — the price of being able to answer "why did the AI do that?" for any decision, ever.

Case study · 04 — the build

The development process is itself an AI deployment

I am not a career engineer. ClaimLayer was built by directing Claude Code, Anthropic's AI coding agent — which makes the build a second exhibit in the same case study: the same questions of specification, trust, and verification, applied to the development loop instead of the claims loop.

📝

Specification over keystrokes

The hard part was never typing the code. It was writing down what to build: the regulatory constraints, the statutory deadlines, the places the model must not be trusted — authored as documents the coding agent works under, the same way the runtime agents work under guardrails.

🔁

Decomposition and review

75+ pull requests, each a reviewable increment with a stated scope. The human checkpoint in the development loop is the same pattern as the human checkpoint in the claims loop: AI drafts, the domain expert decides what ships.

✅

Verification as a guardrail

1,312 tests across 91 suites, including adversarial guardrail tests and schema-contract tests that prove every write shape the code performs. CI applies every migration to a clean database on every push. Trust in AI-written code is earned the same way trust in AI decisions is: by checking, automatically, every time.

The thesis of the whole project, applied twice: AI does the work, the domain expert owns the judgment, code enforces the boundary, and everything is verifiable. That is what I want to do professionally — for products, not just for this one.
